# StarDebate 版本号规范

> 生效日期：2026-06-22

---

## 1. 版本格式

采用 **语义化版本（SemVer）** 三段式：

```
MAJOR.MINOR.PATCH
```

示例：`5.5.0` → `5.5.1` → `5.6.0` → `6.0.0`

### 1.1 各段含义

| 段位 | 名称 | 递增条件 | 复位规则 |
|------|------|----------|----------|
| **MAJOR** | 主版本号 | 重大重构、不兼容的 API 变更、架构级改动 | 重置 MINOR 和 PATCH 为 0 |
| **MINOR** | 次版本号 | 新增功能（向下兼容）、新模块引入 | 重置 PATCH 为 0 |
| **PATCH** | 修订号 | Bug 修复、性能优化、UI 微调（不引入新行为） | 不变 |

### 1.2 初始版本

项目当前版本从 `5.5.0` 开始，后续所有发布均在此基础上递增。

---

## 2. 版本号唯一来源

版本号的 **Single Source of Truth** 为：

```
config/config.json  →  "version" 字段
```

所有需要读取或展示版本号的地方，均应从 `config.json` 读取。

---

## 3. 更新流程

版本号更新通过 **CodeBuddy 聊天交互** 完成，流程如下：

```
完成工作
    ↓
CodeBuddy 询问：「这次改动属于？」
    ├── ① Bug修复 / 小优化  → 递增 PATCH
    ├── ② 新增功能          → 递增 MINOR
    └── ③ 重大变更 / 重构   → 递增 MAJOR
    ↓
CodeBuddy 自动计算新版本号并展示
    ↓
用户确认后，更新 config/config.json
```

### 3.1 更新时机

**每次实质性代码变更后立即更新**，不积攒多次改动后才更新。包括：

- 修复一个 Bug
- 新增一个功能或模块
- 进行重构或架构调整
- 创建更新补丁包时

### 3.2 与补丁包生成的关系

创建更新补丁包（`tools/gen_version_patch.py`）时会 **复用同一流程**：

1. CodeBuddy 先询问版本类型 → 确定新版本号
2. 更新 `config/config.json`
3. 而后以新版本号生成补丁 ZIP（文件名 `update_v{旧版}_to_v{新版}.zip`）

## 4. 版本号使用场景

| 场景 | 来源 | 说明 |
|------|------|------|
| 应用内显示 | `config.config["version"]` | 设置页「关于」、欢迎页等 |
| 补丁包 manifest | 自动从 `config.json` 读取 | `from_version` / `to_version` |
| 补丁包文件名 | 自动生成 | `update_v{旧}_to_v{新}.zip` |
| 插件开发文档 | 人工同步 | `docs/plugin_dev_guide.md` 顶部版本号 |
| 分发文档 | 人工同步 | `docs/update_patch_guide.md` 等 |
