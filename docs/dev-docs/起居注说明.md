## 1. 概述

### 1.1 什么是起居注

**起居注**（ActivityChronicle）是 StarDebate 的自动活动日志系统。它不修改任何现有代码，通过运行时 monkey-patch 自动记录软件中功能运行、插件加载、API 调用、AI 调用的成功与失败。

名称源自中国古代帝王的"起居注"——一种逐日记录帝王言行的史书体裁。在 StarDebate 中，它忠实地记录每一次操作的结果。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| **自动记录** | 无需开发者手动编写日志，启动时自动注入钩子 |
| **不修改源文件** | 通过 monkey-patch 在运行时注入，关闭时恢复原始状态 |
| **成功/失败判定** | 通过拦截 `LogClient.error/warn` 调用自动判断操作是否成功 |
| **嵌套支持** | 子操作错误自动向上传播到父操作 |
| **零开销关闭** | 禁用后所有追踪全部短路，几乎不消耗性能 |

### 1.3 标签

所有起居注日志条目使用统一标签 **`[CRON]`**（Chronicle Record，4 字符），与监视钩子标签体系对齐。

## 3. 核心机制

### 3.1 错误检测：自动判定操作成败

起居注的核心能力是**自动判断一个操作是否成功**。它不需要显式告知"成功"或"失败"，而是通过拦截 `LogClient.error()` 和 `LogClient.warn()` 调用来推断。

```
原理:

  操作开始 → begin("feature", "ai_analysis") → ctx = ChronicleContext()
                                                  ctx.has_error = False
  操作执行 → ... 正常代码 ...

  如果期间任何代码调用了:
    log_client.error("API 超时")   → _on_error("ERROR", ...)
      或
    log_client.warn("参数不完整")   → _on_error("WARN", ...)

  操作结束 → end(ctx, 2345ms) → ctx.has_error = True
                                 → [CRON] ❌ feature·ai_analysis → failed (2345ms): API 超时

  如果期间没有任何 error/warn:
  操作结束 → end(ctx, 2345ms) → ctx.has_error = False
                                 → [CRON] ✅ feature·ai_analysis → ok (2345ms)
```

### 3.2 错误向上传播

当多个操作嵌套时，子操作的错误会向上传播到所有父操作：

```
begin("feature", "pipeline")             # ctx_A
│  begin("api", "step_1")               # ctx_B
│  │  (正常完成)
│  end(ctx_B) → ✅ api·step_1 → ok
│
│  begin("api", "step_2")               # ctx_C
│  │  log_client.error("连接超时")      # → ctx_C.has_error = True
│  │                                    # → ctx_A.has_error = True ★
│  end(ctx_C) → ❌ api·step_2 → failed
│
end(ctx_A) → ❌ feature·pipeline → failed
```

这个机制确保了：**只要流水线中任何一个步骤出错，整个流水线都会被标记为失败**。

### 3.3 异常捕获

通过 `@track` 装饰器或 `track_ctx` 上下文管理器包裹的代码，如果抛出未捕获的异常，起居注同样会标记为失败：

```python
@log_client.track("feature", "dangerous_op")
def dangerous_op():
    raise ValueError("意外的错误")
    # → _chronicle._on_error("ERROR", "意外的错误")
    # → [CRON] ❌ feature·dangerous_op → failed (5ms): 意外的错误
    # → 异常继续向上抛出，不影响业务逻辑
```

### 4.1 四种类别

| 类别 | 标签 | 成功图标 | 开始图标 | 示例 |
|------|------|---------|---------|------|
| `feature` | 功能运行 | ✅ | ▶ | `[CRON] ✅ feature·ai_analysis → ok (2345ms)` |
| `plugin` | 插件加载 | ▶ | ▶ | `[CRON] ▶ plugin·my_timer → ok` |
| `api` | API 调用 | ✓ | ▶ | `[CRON] ✓ api·speech_writer → ok (567ms)` |
| `ai` | AI 调用 | ✅ | ▶ | `[CRON] ✅ ai·call_ai → ok (3456ms)` |

```
成功:
[CRON] {icon} {category}·{name} → ok ({duration}ms)

[CRON]   {icon} {category}·{name} → ok ({duration}ms)
```

## 5. 使用方式

### 5.1 方式 A：装饰器（推荐）

在自定义函数上使用 `@log_client.track()` 装饰器：

```python
@self._log_client.track("feature", "my_analysis")
def run_analysis():
    data = load_data()
    result = process(data)
    if not result:
        self._log_client.error("分析失败：数据为空")  # ← 触发 has_error
    return result

# 正常返回 → [CRON] ✅ feature·my_analysis → ok (1234ms)
# 调用 error() → [CRON] ❌ feature·my_analysis → failed (1234ms): 分析失败：数据为空
# 抛出异常   → [CRON] ❌ feature·my_analysis → failed (5ms): ValueError(...)
```

可以省略 `name` 参数，自动使用函数名：

```python
@self._log_client.track("feature")  # name 自动取 func.__name__
def auto_named_func():
    pass
# → [CRON] ✅ feature·auto_named_func → ok (12ms)
```

### 5.2 方式 B：上下文管理器

在 `with` 块中使用：

```python
with self._log_client.track_ctx("api", "external_upload") as ctx:
    resp = requests.post(url, data=payload)
    if resp.status_code != 200:
        self._log_client.error(f"上传失败: {resp.status_code}")
        # → ctx.has_error = True
    else:
        process_response(resp)

# with 正常退出 → [CRON] ✓ api·external_upload → ok (890ms)
# with 内 error → [CRON] ❌ api·external_upload → failed (890ms): 上传失败: 500
```

### 5.3 方式 C：自动注入（零手动编码）

插件加载/卸载、API 调用、AI 调用在 StarDebate 启动时已自动注入，**无需任何手动代码**。详见 [§6 自动注入钩子](#6-自动注入钩子)。

### 5.4 开关控制

```python
# 启用/禁用起居注
self._log_client.chronicle_enabled = False   # 临时关闭
self._log_client.chronicle_enabled = True    # 重新开启

# 高级：直接访问 Chronicle 实例
chronicle = self._log_client.chronicle
print(f"活跃上下文: {chronicle.active_count}")
```

## 6. 自动注入钩子

### 6.1 注入时机

在 `StarDebateApp._init_chronicle()` 中调用 `install_chronicle(log_client)`，位于管理器实例化之后、`_setup_ui()` 之前。

### 6.2 注入机制

```
install_chronicle(log_client) 执行流程:

  ① 保存原始方法引用
     orig_enable  = PluginInfo.enable
     orig_api     = monitored_api_post
     orig_call_ai = PluginSafeAPI.call_ai

  ② 用 @track 装饰器包装
     PluginInfo.enable = lambda self:
         @log_client.track("plugin", self.name)
         def _run(): return orig_enable(self)
         return _run()

     monitored_api_post = lambda ...:
         @log_client.track("api", feature_name)
         def _run(): return orig_api(...)
         return _run()

     PluginSafeAPI.call_ai = lambda self, ...:
         @log_client.track("ai", "call_ai")
         def _run(): return orig_call_ai(self, ...)
         return _run()

  ③ 返回 saved_refs 供 closeEvent 中 uninstall 恢复
```

### 6.3 覆盖范围

| 注入目标 | 类别 | 名称 | 触发时机 |
|---------|------|------|---------|
| `PluginInfo.enable()` | `plugin` | 插件名称 | 用户开启/导入插件 |
| `PluginInfo.disable()` | `plugin` | 插件名称 | 用户关闭插件 |
| `monitored_api_post()` | `api` | feature_name (如 `ai_analysis`) | 任何 AI Worker 的 HTTP 请求 |
| `PluginSafeAPI.call_ai()` | `ai` | `call_ai` | 插件通过 `api.call_ai()` 调用 AI |

### 6.4 不修改的文件

以下源文件**完全没有改动**：

- `workers/plugin_manager/__init__.py`
- `workers/common/api_helper.py`
- `workers/plugin_manager/plugin_api.py`
- `workers/debug_console/debug_monitor_manager.py`

## 7. 嵌套操作支持

### 7.1 上下文栈

`ActivityChronicle` 内部维护一个 **操作上下文栈**（`_stack: list[ChronicleContext]`），支持任意深度的嵌套：

```
begin("feature", "export_all")        _stack = [ctx_A(depth=0)]
  ├─ begin("feature", "collect_data")  _stack = [ctx_A, ctx_B(depth=1)]
  │   └─ end(ctx_B)                    _stack = [ctx_A]
  ├─ begin("api", "upload_chunk_1")    _stack = [ctx_A, ctx_C(depth=1)]
  │   └─ end(ctx_C)                    _stack = [ctx_A]
  ├─ begin("api", "upload_chunk_2")    _stack = [ctx_A, ctx_D(depth=1)]
  │   │  log_client.error("失败")      → ctx_D.has_error = True
  │   │                                → ctx_A.has_error = True ★
  │   └─ end(ctx_D)                    _stack = [ctx_A]
  └─ end(ctx_A)                        _stack = []
                                        → ❌ failed (ctx_A.has_error = True)
```

### 7.2 缩进显示

嵌套操作的日志输出带有自动缩进，清晰展示层级关系：

```
[CRON] ▶ feature·export_all start
[CRON]   ✅ feature·collect_data → ok (45ms)
[CRON]   ✓ api·upload_chunk_1 → ok (1234ms)
[CRON]   ❌ api·upload_chunk_2 → failed (567ms): 网络错误
[CRON] ❌ feature·export_all → failed (1900ms): 网络错误
```

### 7.3 错误传播规则

| 场景 | 行为 |
|------|------|
| 子操作调用 `log_client.error()` | 子操作 `has_error=True`，所有父操作 `has_error=True` |
| 子操作调用 `log_client.warn()` | 同上 |
| 子操作抛出异常未捕获 | 由装饰器/上下文管理器捕获，标记 `has_error`，异常继续抛出 |
| 父操作自身调用 `log_client.error()` | 父操作 `has_error=True`（不影响已完成子操作） |

## 8. 配置说明

### 8.1 配置文件

**路径**：`config/chronicle_config.json`

```json
{
    "enabled": true,
    "categories": {
        "feature": true,
        "plugin": true,
        "api": true,
        "ai": true
    },
    "min_duration_ms": 0,
    "log_level": "INFO"
}
```

### 8.2 字段说明

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enabled` | `bool` | `true` | 起居注总开关。设为 `false` 后所有追踪失效 |
| `categories.feature` | `bool` | `true` | 功能运行追踪（`@track("feature",...)`) |
| `categories.plugin` | `bool` | `true` | 插件加载/卸载追踪 |
| `categories.api` | `bool` | `true` | API 调用追踪 |
| `categories.ai` | `bool` | `true` | AI 调用追踪 |
| `min_duration_ms` | `int` | `0` | 最短记录时间。低于此耗时的操作不写入日志 |
| `log_level` | `str` | `"INFO"` | 起居注日志的写入级别 |

### 8.3 运行时开关

除了配置文件，还支持运行时动态控制：

```python
# 总开关（配置 + 运行时双重控制）
self._log_client.chronicle_enabled = False

# 等价于:
self._log_client.chronicle.enabled = False
```

运行时开关优先级高于配置文件：配置文件 `enabled=true` 但运行时设为 `false` 时仍不生效。

### 8.4 性能影响

| 状态 | 开销 | 说明 |
|------|------|------|
| `enabled=true` | 极低 | `begin/end` 各约 0.01ms；`error/warn` 多一次栈遍历 |
| `enabled=false` | 几乎零 | `begin` 返回哑上下文，`_on_error` 短路跳过 |
| 分类关闭 | 单独零 | 该类别 `begin` 返回哑上下文，不影响其他类别 |

## 9. 与监视钩子的区别

起居注 `[CRON]` 和调试监视钩子 `[VAR/FUNC/API/AI/PLUGIN]` 是两个正交系统：

| 维度 | 起居注 `[CRON]` | 监视钩子 `[VAR/FUNC/...]` |
|------|----------------|--------------------------|
| **目的** | 记录"做了什么、成功与否" | 记录"内部发生了什么" |
| **粒度** | 粗粒度（操作级别） | 细粒度（变量/函数/请求） |
| **自动性** | 全自动（monkey-patch 注入） | 需手动插入 + 调试台开启 |
| **输出** | 摘要：成功/失败+耗时 | 详情：值变化/调用栈/响应体 |
| **开关** | 配置文件控制 | 调试台「调试模式」控制 |
| **典型条目** | `✅ feature·export → ok (1.2s)` | `[API] POST /v1/chat → 200 1.2s` |
| **崩溃安全** | 通过 LogService 独立进程 | 通过 LogService 独立进程 |
| **修改源文件** | 不修改 | 需要 |

**两者互补**：起居注提供宏观视图（"导出功能成功了"），监视钩子提供微观调试（"API 请求耗时 1.2s，响应 200"）。

## 10. 常见问题

### Q: 起居注会影响性能吗？

几乎不影响。`enabled=true` 时 `begin()/end()` 各约 0.01ms；`enabled=false` 时全部短路跳过，零开销。

### Q: 起居注和监视钩子日志会冲突吗？

不会。它们使用不同的标签（`[CRON]` vs `[VAR/FUNC/...]`），可以同时在同一日志文件中输出，互不干扰。

### Q: 如何临时关闭起居注？

```python
self._log_client.chronicle_enabled = False
```

或修改 `config/chronicle_config.json` 中 `"enabled": false`。

### Q: 如何关闭某类别（例如只追踪 API，不追踪插件）？

修改 `config/chronicle_config.json`：
```json
{
    "categories": {
        "feature": false,
        "plugin": false,
        "api": true,
        "ai": true
    }
}
```

### Q: 装饰器和上下文管理器如何选择？

| 场景 | 推荐方式 |
|------|---------|
| 整个函数需要追踪 | `@track` 装饰器 |
| 函数内某段代码需要追踪 | `with track_ctx()` |
| 需要访问 `ctx` 对象 | `with track_ctx() as ctx` |
| 已注入的自动钩子 | 无需选择，自动生效 |

### Q: 嵌套操作中，如果子操作失败，父操作一定会失败吗？

是的。只要子操作期间有任何 `log_client.error()` 或 `log_client.warn()` 调用（或未捕获异常），父操作就会被标记 `has_error=True`。

### Q: 起居注会记录 python `print()` 的输出吗？

不会。起居注只拦截 `LogClient.error()` 和 `LogClient.warn()` 调用。`print()` 和 `sys.stderr` 不受起居注监控。

### Q: 如果 LogService 进程崩溃，起居注数据会丢吗？

不会。起居注的日志条目通过 `LogClient._send()` 发送到队列。如果队列不可用，会触发应急降级 `_emergency_write()` 直写文件。详见 [日志系统独立进程架构说明]。

### Q: monkey-patch 注入会不会有副作用？

不会。所有被包装的原始方法在 `install_chronicle()` 时保存，在 `closeEvent` 中通过 `uninstall_chronicle()` 完全恢复。包装层仅添加 `@track` 装饰器，不影响原始逻辑。

### Q: 日志文件什么时候会被删除？

**主窗口正常退出时，自动删除日志文件**。崩溃（包括进程被杀死、段错误、未捕获异常退出）时日志文件保留，用于事后诊断。这个逻辑在 `StarDebate.py` 启动器中实现：

```
正常退出: closeEvent → shutdown → LogService.join() → os.remove(log_path)
崩溃退出: LogService 检测主进程消失 → 保留日志 + 写入 [SYS] 崩溃快照
```

## 11. ★ v2.0.0 崩溃定位增强

### 11.1 为什么需要增强

v1.0.0 起居注能追踪操作的"成功"或"失败"，但面对难以发现的崩溃（如段错误、Qt 内部 C++ 异常、Worker 线程静默退出），传统日志往往不完整。v2.0.0 新增以下功能：

### 11.2 六大增强

| 功能 | 说明 | 解决场景 |
|------|------|----------|
| **完整 traceback** | 异常捕获保留调用栈（`_on_exception`） | Python 层崩溃的精确代码定位 |
| **Lineage 快照** | 每 2 秒将活跃操作链写入共享状态 | 崩溃时知道"正在做哪一步" |
| **守护日志环** | LogClient 内存环形缓冲区（200 条） | 崩溃前最后几毫秒的日志不丢失 |
| **Qt message handler** | 拦截 qWarning/qCritical/qFatal | Qt C++ 层内部错误 |
| **Worker 线程保护** | patching 所有 AI Worker 的 `run()` 方法 | QThread 中的未处理异常 |
| **崩溃自动分析** | `SystemHealthMonitor` 分析崩溃前日志输出诊断摘要 | 减少人工排查时间 |

### 11.3 崩溃快照输出示例

正常退出时，起居注快照随 `emergency_info` 发送到 LogService；崩溃时，`SystemHealthMonitor` 在命令行快照中额外写入：

```
═══ [SYS] 主进程异常退出 ═══
  崩溃时活跃操作链: feature·export → api·speech_writer → ai·call_ai
  活跃上下文数量: 3
    [0] feature·export (depth=0) [运行中] 已运行15678ms 线程:123
       └ 元数据: debate_path=e:\StarDebate\test, format=标准赛制
    [1] api·speech_writer (depth=1) [⚠ 已报错] 已运行12345ms
       └ API 调用超时: https://api.example.com/chat
    [2] ai·call_ai (depth=1) [运行中] 已运行3456ms

═══ [RING] 守护日志环 (崩溃前最近 200 条) ═══

═══ 💡 崩溃自动分析 ═══
  1. 🔴 未完成的操作 (3 个): feature·export, api·speech_writer, ai·call_ai
  2. 🟡 崩溃前 5秒内 error/warn: 2 条
  3. 🟡 崩溃时 4 个线程活跃，可能为多线程竞争
  ─── 建议 ───
  → 检查未完成操作相关的代码路径，优先排查该函数的异常处理
  → 检查 Worker 线程的生命周期管理，可能存在线程泄露或竞争条件
```
