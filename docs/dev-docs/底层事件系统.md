## 1. 概述

**底层事件记录系统**（Native Event Logging）是 StarDebate 日志系统的底层补充。与起居注 `[CRON]`（记录"什么操作成功/失败"）不同，底层事件系统关注**更深层的信号源**：

| 层级 | 信号源 | 标签前缀 | 里程碑 |
|------|--------|---------|--------|
| L1 | Qt C++ 层 (qFatal/qCritical/qWarning) | `[NATIVE]` | M1 ✅ |
| L2 | Python excepthook (未捕获异常) | `[NATIVE]` | M1 ✅ |
| L2 | Python unraisablehook (`__del__`/生成器/`__exit__` 异常) | `[NATIVE]` | M2 ✅ |
| L2 | Python audithook (import/open/socket/ctypes) | `[NATIVE]` | M2 ✅ |
| L3 | GC 回收异常 (gc.callbacks) | `[RES]` | M2 ✅ |
| L2 | c_exception (C 函数异常, sys.setprofile, 默认关闭) | `[NATIVE]` | M2 ✅ |
| L4 | 资源跟踪 (fd 统计/Qt 无父 widget) | `[RES]` | M3 📅 |
| L5 | 线程健康 (死检测/卡死/死锁/主循环心跳) | `[THREAD]` | M3 📅 |
| L6 | 第三方库错误 (requests/ctypes/json) | `[EXT]` | M4 ✅ |

### 1.1 核心设计

```
双轨制落地方案：
┌─ A: 文本主日志 → docs/log/debug_*.log（[NATIVE] 标签，人类可读）
└─ C: SQLite     → docs/log/native.db（3 表 WAL 模式，结构化存储）
```

### 1.2 目标

- 捕获起居注 `[CRON]` 无法触及的底层异常
- 每条事件同时写入文本日志（快速查看）和 SQLite（深入分析）
- 彻底解决"无声崩溃"（无日志前兆的段错误/死锁/静默卡死）

### 2.2 写入流程

```
钩子触发 → write_event()
              │
              ├─ 节流检查（同一 source 每秒 >50 则丢弃）
              │
              ├─ 入队列 _event_queue ← 线程安全 queue.Queue
              │
              ├─ _batch_writer_loop (守护线程)
              │    └─ 每 200ms 或满 100 条
              │         └─ _flush_sqlite(batch)
              │              ├─ INSERT ... VALUES ... (逐条)
              │              └─ conn.commit()
              │
              └─ _write_text_log() → LogClient.info/warn/error
                   └─ 输出到现有 debug_*.log 文件
```

## 3. 数据库设计

### 3.1 native.db 三张表

#### 表 1：`native_events`（Qt/Python 原生异常源）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | 自增主键 |
| `ts` | REAL | 事件时间戳 (`time.time()`) |
| `level` | TEXT | 级别 (`qt_fatal`/`qt_critical`/`qt_warning`/`uncaught`/`unraisable`/`audit`/`thread`) |
| `source` | TEXT | 信号源标识 (`QtMsgHandler`/`sys.excepthook`/`sys.unraisablehook`/`sys.addaudithook`) |
| `location` | TEXT | 定位串 (`qwidget.cpp:L4832:QWidget::setParent`) |
| `message` | TEXT | 主消息（截断 500 字符） |
| `detail_json` | TEXT | 结构化补充（traceback、context、event args 等，JSON 编码，截断 2000 字符） |
| `thread_id` | INTEGER | 线程 ID |
| `thread_name` | TEXT | 线程名（截断 50 字符） |
| `func_name` | TEXT | 当前函数名（取自 traceback 帧顶部，截断 100 字符） |

索引：`(ts)`, `(level)`, `(source)`

#### 表 2：`thread_events`（线程健康）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | 自增主键 |
| `ts` | REAL | 时间戳 |
| `kind` | TEXT | 事件种类 (`thread_died`/`thread_stuck`/`deadlock_suspect`/`qt_loop_blocked`/`main_loop_no_resp`) |
| `thread_id` | INTEGER | 线程 ID |
| `thread_name` | TEXT | 线程名 |
| `last_func` | TEXT | 最后一次活动函数 |
| `stuck_seconds` | REAL | 卡死时长（秒） |
| `detail` | TEXT | 详细说明 |

索引：`(ts)`, `(kind)`

#### 表 3：`resource_events`（资源问题）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | 自增主键 |
| `ts` | REAL | 时间戳 |
| `kind` | TEXT | 种类 (`fd_leak`/`socket_leak`/`qt_object_no_parent`/`qt_object_leak`/`gc_uncollectable`) |
| `obj_type` | TEXT | 对象类型名 |
| `obj_repr` | TEXT | 对象 repr（截断） |
| `count` | INTEGER | 当前累积数量 |
| `detail` | TEXT | 详情 |

索引：`(ts)`, `(kind)`

### 3.2 存储策略

| 参数 | 值 |
|------|-----|
| journal_mode | `WAL`（Write-Ahead Logging，崩溃安全） |
| synchronous | `NORMAL`（保持性能 + 合理持久性） |
| busy_timeout | 3000ms |
| 单表上限 | 50000 行 |
| 保留时间 | 7 天（启动时自动清理过期数据） |

### 3.3 清理逻辑

每次 `close()` 时执行：
1. `DELETE FROM <表> WHERE ts < (now - 7 天)` — 过期数据
2. `DELETE FROM <表> WHERE id NOT IN (SELECT id FROM <表> ORDER BY id DESC LIMIT 50000)` — 超量数据

## 4. 钩子系统

### 4.1 M1 钩子（已实现）

| 钩子 | 层级 | 安装位置 | 说明 |
|------|------|---------|------|
| Qt message handler | L1 | `native_hooks._install_qt_hook()` | 拦截 qWarning/qCritical/qFatal，提取完整 QMessageLogContext（文件/行/函数/类别），写入 SQLite，然后调用原始 handler |
| excepthook | L2 | `native_hooks._install_excepthook()` | 拦截未捕获 Python 异常，提取完整 traceback，写入 SQLite，然后调用原始 hook |

### 4.2 钩子链式调用

```
原始 sys.excepthook = sys.__excepthook__（Python 默认）
     ↓  LogClient._install_global_hooks() 注册
sys.excepthook = _chronicle_excepthook（起居注，写 [CRON] + 守护日志环）
     ↓  native_hooks._install_excepthook() 注册
sys.excepthook = _native_excepthook（底层事件，写 SQLite + [NATIVE]）
     ↓ 调用原始 hook
_chronicle_excepthook（起居注）
     ↓ 调用原始 hook
sys.__excepthook__（默认，写 stderr）
```

### 4.3 M2 钩子（已实现）

| 钩子 | 层级 | 安装位置 | 说明 |
|------|------|---------|------|
| `sys.unraisablehook` | L2 | `native_hooks._install_unraisablehook()` | 捕获 `__del__`/生成器 `.throw()`/`__exit__` 等不可抛出异常。这些是 `sys.excepthook` 抓不到的盲区。提取异常类型 + 源对象 repr + 完整 traceback → SQLite |
| `sys.addaudithook` | L2 | `native_hooks._install_audithook()` | 捕获 import/open/socket/ctypes/subprocess 等安全敏感审计事件。白名单过滤（仅关注约 15 种事件），参数摘要记录。注意：`sys.addaudithook` 是 append-only，无法移除；关闭时通过 `manager.is_running` 标志软跳过 |
| `gc.callbacks` | L3 | `native_hooks._install_gc_callbacks()` | GC 周期回调，仅关注 `"stop"` 阶段：`uncollectable > 0`（不可回收对象检测）和 `collected == 0`（可疑循环引用）。记录类型分布和 GC 统计 |
| `sys.setprofile` (c_exception) | L2 | `native_hooks._install_c_exception_profile()` | 捕获 C 函数异常。**默认关闭**（`hooks.c_exception_profile: false`），因 `sys.setprofile` 在每次函数调用/返回时触发，有显著性能开销。启用后消噪：同源类型每秒最多 5 条。采样局部变量 |

### 4.4 M3 组件（已实现）

| 组件 | 层级 | 文件 | 说明 |
|------|------|------|------|
| 线程健康监视器 | L5 | `native_thread_monitor.py` | 4 类检测：线程消失 (10s/次) / 卡死 (5s/栈帧 30s 不变) / 死锁嫌疑 (30s/Lock.acquire 60s) / Qt 主循环心跳 (3s/5s 无响应)。独立守护线程，`sys._current_frames()` 帧签名比对 |
| 资源跟踪监视器 | L4 | `native_resource_monitor.py` | fd 句柄计数（Windows GetProcessHandleCount，60s/次，阈值 1000）+ Qt 孤立 Widget 扫描 + atexit 退出检查 |
| 起居注桥接器 | - | `native_chronicle_bridge.py` | 双向弱联动：包装 `_write_text_log`，deadlock_suspect/main_loop_no_resp → 始终桥接 ERROR；fd_leak > 2000 → 条件桥接 ERROR |

### 4.5 线程健康检测细节

线程健康监视器运行在独立守护线程中：

```
NativeThreadMonitor._run()
  │
  ├─ _scan_thread_alive (每 10s)
  │    threading.enumerate() 枚举 → 对比 _known_threads
  │    → 消失的线程写 thread_died event
  │
  ├─ _scan_thread_stuck (每 5s)
  │    sys._current_frames() 取帧 → 计算帧签名（10 层栈）
  │    → 签名 30s 不变写 thread_stuck event（跳过主线程）
  │
  ├─ _scan_deadlock_suspect (每 30s)
  │    sys._current_frames() 遍历 → 沿调用栈查找 Lock.acquire
  │    → 同一位置 60s 不变写 deadlock_suspect event
  │
  └─ _check_qt_heartbeat (每 3s)
       QTimer 每秒更新 manager 时间戳
       → 5s 未更新写 main_loop_no_resp event
```

### 4.6 起居注桥接条件

| 事件 | 级别 | 触发条件 | 桥接行为 |
|------|------|----------|----------|
| `deadlock_suspect` | ERROR | 始终 | `chronicle._on_error("ERROR", msg)` → 标记所有活跃 CRON context |
| `main_loop_no_resp` | ERROR | 始终 | ↑ |
| `fd_leak` | ERROR | count > 2000 | ↑（count ≤ 2000 时不桥接，仅在文本日志记录） |
| 其他 | - | 不桥接 | 仅文本日志 + SQLite |

### 4.8 安全保护

所有钩子有 3 重保护：
1. **入口 try/except** — `except Exception: pass`，绝不向上抛出
2. **递归保护** — `threading.local()` 标志位，同一线程重复进入时跳过
3. **启动健康检查** — 30 秒后自检，写入测试事件并回读验证

## 7. 与起居注的关系

底层事件系统和起居注是两个正交系统，**互补而非替代**：

| 维度 | 起居注 `[CRON]` | 底层事件 `[NATIVE]` |
|------|----------------|--------------------|
| **目的** | 追踪操作的成功/失败 | 捕获底层异常信号 |
| **粒度** | 操作级别（如"导出"、"分析"） | 信号源级别（如"Qt 内存错误"） |
| **自动性** | monkey-patch 注入 | 原生钩子注册 |
| **输出** | `[CRON] ✅/❌ category·name` | `[NATIVE] [level] location: msg` |
| **数据模型** | 上下文栈 + 失败传播 | SQLite 3 表结构化 |
| **覆盖** | 插件/API/AI 调用 + Worker 线程 | Qt 内部 / 全局 hook / 线程 / 资源 |
| **崩溃后** | 起居注 lineage 快照 | SQLite WAL 崩溃安全恢复 |

**双向弱联动**（M3 实施）：
- 仅"真会引发故障"的底层事件（如 `deadlock_suspect`、`fd_leak` 超阈值）自动注入起居注上下文
- 起居注失败标记时自动查询底层事件数据库作为诊断上下文

## 8. 性能与鲁棒性

### 8.1 性能预算

| 维度 | 预算 | 保护 |
|------|------|------|
| 稳态事件速率 | ≤ 5 条/秒 | 节流（≥ 50/s 自动丢弃） |
| 崩溃峰值速率 | ≤ 100 条/秒 | 队列缓冲，不阻塞主线程 |
| SQLite 写入延迟 | ≤ 5ms/条（批处理后） | 独立线程 + 批量提交 |
| 内存占用 | ≤ 1MB（队列 ≤ 1000 条） | 满则强制 flush |
| 钩子自身开销 | ≤ 0.01ms/次 | try/except 短路 |

### 8.2 安全机制

```
系统级别:
  - 总开关 enabled=false → 所有钩子跳过，零开销
  - 类别级开关 hooks.*=false → 单个钩子跳过

钩子级别:
  - try/except → 绝不让钩子自身异常蔓延
  - 递归保护 → threading.local() 标志位防止重复进入
  - 启动健康检查 → 30s 后自检，失败则自动降级

数据库级别:
  - WAL journal_mode → 写入时主进程崩溃不丢数据
  - busy_timeout=3000 → 避免 SQLite 锁冲突
  - 自动清理 → 数据量不无限增长

降级路径:
  - 队列满 → 强制 flush
  - SQLite 不可写 → 只写文本日志
  - 文本日志不可写 → 静默丢弃（已有 LogService 保证）
```

## 10. 常见问题

### Q: 底层事件系统和起居注有什么区别？

起居注记录"操作成功/失败"（粗粒度、摘要式）。底层事件记录"Qt 内部错误 / Python 未捕获异常 / 线程卡死"（细粒度、信号源式）。两者互补。

### Q: SQLite 数据库会不会撑爆磁盘？

不会。单表上限 50000 行（约 20MB），保留 7 天。每次 `close()` 自动清理。

### Q: 底层事件系统会影响性能吗？

稳态下零影响。所有写入通过队列异步，不阻塞主线程。节流阈值 50/s，低价值事件自动丢弃。

### Q: 如果我关闭了底层事件系统，起居注还正常工作吗？

是的。两个系统完全独立。`enabled=false` 时所有底层钩子不注册，起居注不受影响。

### Q: 优化建议

**文本日志级别调整**：如果 `[NATIVE]` 标签在文本日志中过于频繁，可将 `text_log_level` 设为 `WARN`（仅 ERROR/WARN 写入文本日志）。

**节流阈值调整**：如果某个来源（如 Qt warning）在短时间内触发大量同类事件，可调低 `max_per_second_per_category`。
